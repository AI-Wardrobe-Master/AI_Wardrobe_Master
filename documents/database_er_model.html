1. 建模范围

本文档把项目里的数据库内容分成两类：

  • 已实现并进入当前后端 Schema 的表：users、clothing_items、images、models_3d、processing_tasks、wardrobes、wardrobe_items。
  • 文档中规划过但当前后端未落地的实体：user_profiles、outfits、outfit_items、creators、card_packs、card_pack_items、import_history、provenance。
说明：documents/DATA_MODEL.md 和 documents/BACKEND_ARCHITECTURE.md 包含较完整的产品级概念模型，但仓库中的后端 ORM / Alembic / init_schema 目前只实现了衣物、图片、3D 模型、处理任务和衣橱管理这条主链路。

4. 实体清单

下面是当前后端已落地表的“逻辑意义 + 物理字段 + 关键约束”摘要，便于你在做评审或对齐接口时直接查看。

users

  • 用户根表，承载认证与归属关系。
  • 字段：id、username、email、hashed_password、user_type、is_active、created_at、updated_at。
  • 唯一约束：username、email。

clothing_items

  • 项目的中心业务实体，所有媒体、模型、任务、衣橱挂载都围绕它展开。
  • 字段包含 JSONB 标签、自由标签数组、确认状态和基础元信息。
  • source 区分 OWNED / IMPORTED，当前实现里是衣物主记录的入口。

images

  • 保存衣物原图、背景移除后的图、角度图，数据库只存路径，文件本体放对象存储。
  • image_type 用于区分 ORIGINAL_FRONT、ORIGINAL_BACK、PROCESSED_FRONT、PROCESSED_BACK、ANGLE_VIEW。
  • 角度图使用 0/45/90/135/180/225/270/315 八个固定角度。

models_3d

  • 存储 3D 模型元数据与模型文件路径，当前模型格式固定为 glb。
  • 对 clothing_item_id 做唯一约束，所以一件衣物只有一个 3D 模型记录。
  • 用于承接三维重建流程的输出。

processing_tasks

  • 异步流水线任务表，记录分类、背景移除、3D 生成、角度渲染和整条流水线的执行状态。
  • 当前实现新增 attempt_no、retry_of_task_id、worker_id、lease_expires_at，用于重试和 worker 租约控制。
  • 对 active task 做了 partial unique index，避免同一衣物同时存在多个 PENDING / PROCESSING 任务。

wardrobes

  • 用户衣橱容器表，支持 REGULAR / VIRTUAL 两种类型。
  • 删除衣橱不会删除衣物本体，只会删除关联行。
  • 是衣物集合管理的组织层。

wardrobe_items

  • 衣橱与衣物之间的多对多连接表。
  • display_order 允许前端控制排序，added_at 记录加入时间。
  • 唯一组合键防止同一衣物重复出现在同一个衣橱内。

6. 设计结论

  • clothing_items 是中心枢纽：衣物是所有媒体、任务和衣橱组织关系的根。
  • images / models_3d / processing_tasks 属于衣物的派生域：它们都依附于衣物记录，且以衣物 ID 作为主外键。
  • wardrobes 只是容器：它不复制衣物数据，而是通过 wardrobe_items 建立可排序的集合关系。
  • processing_tasks 采用任务历史化设计：重试不是覆盖旧记录，而是生成新任务，并通过 retry_of_task_id 保留链路。
  • 当前实现与产品草案是分层的：文档里更大的用户/创作者/搭配体系还没有进入后端物理 schema。
