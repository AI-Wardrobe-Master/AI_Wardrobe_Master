## BLJ 可视化模块工作说明 2026.3.6

### 1. 负责范围

本次主要完成 `Visualize` 页面及其相关交互，目标是把原本较为基础的占位式可视化页面，提升为一个可以进行服饰选择、分层穿搭、预览生成、图片保存的完整前端演示流程。

需要说明的是，我们负责的是项目中的一部分功能。本次提交后，是否可以认定为该部分正式完成，仍需等待项目经理审查。

### 2. 本次新增的核心功能

#### 2.1 可视化主画布重构

- 将原来的灰色占位模特替换为全身参考照片画布。
- 在画布上保留四个身体区域入口：
  - `Head`
  - `Upper Body`
  - `Lower Body`
  - `Feet`
- 每个区域均通过半透明的 `+` 热点按钮进入选衣流程。
- 画布支持在人物对应区域展示已穿上的服饰状态，而不是只停留在空白占位。

#### 2.2 分区服饰选择能力

- 为四个身体区域分别建立了可选服饰目录。
- `Head` 与 `Feet` 采用单件穿戴逻辑。
- `Upper Body` 与 `Lower Body` 支持分层穿搭逻辑。
- 分层规则目前为最多 3 层，便于表达：
  - 贴身层
  - 中间层
  - 外层

#### 2.3 服饰卡片交互重构

为减少交互歧义，对底部选择器中的服饰卡片做了重新设计：

- 点击卡片正文：进入衣物详情入口。
- 点击 `Wear` 绿色按钮：执行穿上逻辑。
- 点击 `Remove` 红色按钮：执行脱下逻辑。

相比之前的 `Actions` 入口，这种方式更直接，也更符合移动端操作习惯。

#### 2.4 分层穿搭交互

对于上衣和下装：

- 点击 `Wear` 后不会直接穿上。
- 系统会弹出一个单独的层级选择面板。
- 用户可明确选择穿在：
  - `Layer 1`
  - `Layer 2`
  - `Layer 3`
- 如果目标层已有衣物，界面会提示替换关系。

这样可以更准确地表达多层服饰的组合关系，而不是简单覆盖。

#### 2.5 已穿服饰标签展示优化

针对画布上的“已穿标签”做了多轮优化：

- 将过大的横向标签缩小为更紧凑的展示形式。
- 上衣和下装的已穿标签改为可堆叠展示，能够看出当前穿了几层。
- 标签支持点击，点击后可进入衣物详情入口。
- 标签位置重新调整，尽量避免遮挡 `+` 热点。
- 标签文案从容易引起误解的 `Active` 改为 `Wearing`。

这一部分目前已经明显优于初始版本，但在极端机型下若仍有局部遮挡，后续还可以继续做更细的分区坐标微调。

#### 2.6 衣物详情入口预留

目前已经预留“点击卡片正文/点击已穿标签进入衣物详情”的流程。

当前状态：

- 已有详情弹窗入口；
- 能显示该衣物的基础信息；
- 能展示该衣物当前是否正在穿着；
- 能展示所属身体区域、材质、描述等信息。

但需要说明：

- 这还不是最终的完整衣物详情页面；
- 目前更像是为后续详情模块预留的过渡入口。

#### 2.7 预览生成流程

在选择好衣物后，页面底部提供 `Generate Preview` 按钮。

流程如下：

1. 用户完成穿搭选择；
2. 点击 `Generate Preview`；
3. 弹出预览弹窗；
4. 弹窗中展示生成效果图；
5. 用户可选择：
   - `Close`
   - `Save to Gallery`

这里的“生成预览”目前使用的是预置好的示例预览图资源，用于演示完整交互流程。它证明了页面流程、弹窗逻辑和保存能力已经打通，但并不代表后端真实生成逻辑已经接入。

#### 2.8 清空逻辑

页面底部保留 `Clear` 按钮。

点击后会：

- 清空当前所有已选服饰；
- 清空分层穿搭状态；
- 回到原始全身参考照片状态。

这样方便用户重新开始新一轮搭配，而不必逐件手动移除。

#### 2.9 保存到手机相册

已经完成 Android 侧原生保存逻辑接入。

当前效果：

- 在预览弹窗中点击 `Save to Gallery`；
- Flutter 通过 `MethodChannel` 调用 Android 原生代码；
- 原生侧使用 `MediaStore` 将图片写入系统相册目录；
- 用户可以在手机图库中直接看到保存结果。

这意味着我们不仅完成了前端 UI，还完成了“从界面到系统相册”的完整落地链路。

### 3. 界面与文案调整

除了主功能外，还针对可用性做了一批细化调整：

- 所有新增代码与界面文案均使用英文；
- 说明文案中明确区分“查看详情”和“执行穿/脱操作”；
- 材质标签从卡片底部移动到更靠上的位置，使信息层级更清晰；
- 状态标签统一使用 `Wearing`，减少误解；
- `Wear` 与 `Remove` 按钮使用明显不同的颜色，降低误触和认知负担。

### 4. 适配与稳定性优化

为了提高不同 Android 设备上的可用性，本次还补充了多项适配处理：

- 底部弹窗使用受控高度，防止内容过长导致溢出；
- 长内容区域支持滚动；
- 服饰卡片在窄屏下会切换为更适合的小屏布局；
- 预览弹窗会根据宽度自动调整按钮排布；
- 选择区与操作区均针对小屏手机进行了测试覆盖。

虽然无法一次性保证“任何未来设备”都完全零问题，但目前代码已经从固定布局提升为较强的响应式布局。

### 2. 关于“Generate Preview”是否已接入真实 AI 生成

这件事已经重新核对过，结论必须明确写清：

- **当前前端点击 `Generate Preview` 后，并不会向后端请求一张真实生成的 outfit preview 图。**
- 当前显示的仍然是本地静态资源：
  - `assets/visualization/preview/generated_outfit_preview.jpg`
- 因此它目前仍属于 **demo preview**，不能表述为“已经实现 AI 生成穿搭预览”。

### 3. 关于后端是否已有“可直接调用的可视化模型”

本次还专门检查了后端现状，结论如下：

- 后端现在确实已经有 **单件衣物** 的处理链路：
  - 去背景
  - 自动分类
  - 3D 模型生成
  - 多角度渲染
- 但后端 **还没有真正实现 outfit / try-on / visualization preview** 的 API 路由。
- 也就是说，目前后端的模型能力更接近“把一件衣服数字化”，而不是“把多件衣物穿到人体上生成整体预览图”。

因此，从项目交付角度必须区分：

- **单件衣物 3D/多视角处理**：已有后端基础；
- **整套穿搭真实预览生成**：当前仍未完成。

### 4. 本次新增功能：Outfit Collection（穿搭集合）

为了让可视化模块不只停留在“试穿一下”，本次新增了一个独立于真实衣柜的数据结构：

- `Outfit Collection`

它与普通衣柜不同，用于保存一套完整的穿搭组合，当前已支持：

- 从可视化预览弹窗中导出当前已选穿搭；
- 为该穿搭填写：
  - collection name
  - tags
  - how to wear
- 保存所选衣物内容与分层信息；
- 记录预览参考图；
- 自动生成 share code / share link；
- 在 `Discover` 页面中查看已导出的 collection。

这个结构更适合后续继续扩展为：

- 创作者分享的穿搭卡包；
- 用户自己的穿搭收藏；
- 可导入/可分享的 look 数据。

### 5. Demo 展示补强

- 因此这部分应理解为 demo 层面的视觉补位，而不是后端真实图片生成能力。

### 8. 4 月 5 日最终稳定性修复补记

在完成 `Outfit Collection` 基本功能后，又继续处理了两项影响演示体验的问题：

- 导出页曾同时在顶部和底部放置 `Save Collection`，造成重复操作入口；
- 真机点击保存时曾触发 Flutter framework 红屏断言，影响演示稳定性。

本轮已完成修复：

- 将导出页顶部重复的 `Save Collection` 按钮移除，仅保留底部主操作按钮；
- 将导出页状态管理整理为独立页面状态，减少保存时的生命周期冲突；
- 标签输入改为真正的多标签模式，可一次输入多个标签，而不是只能写成一整段；
- 在连接的一加真机上重新走通完整链路并确认：
  - `Visualize`
  - `Head Picker`
  - `Wear`
  - `Confirm`
  - `Generate Preview`
  - `Export to Collection`
  - `Save Collection`
- 本次实测结果为：
  - 不再出现保存时红屏；
  - 保存后会正常返回预览弹窗；
  - 新集合会在 `Discover` 页面出现。

因此，截至本次修复，可视化模块中的“导出穿搭集合”演示链路已经达到可稳定展示的状态。
